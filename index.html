GDPVal Pilot Program

Every task is authored by a domain expert drawing from real professional work pattern. Each task is modelled on an actual professional assignment: a board memo requiring a cost bridge, a feasibility study requiring capacity analysis, an investment decision requiring financial modelling.

The benchmark covers Management Consulting (O*NET 13-1111.00, Management Analysts), with the six tasks here spanning operations optimization, cost optimization, operations research, and market strategy.

Six expert-authored management-consulting tasks, each run three times on Qwen 3.6 27B and scored by domain SMEs against atomic, DAG-weighted verifiers. This showcase presents every task in full — prompt, hidden trap, golden trajectory and deliverable, the verifier rubric, and the dependency graph — alongside how the model actually performed.

01Anatomy of a task page

Each task page carries the six golden-sample components. Inputs (prompt + reference files) are shown to the model; the ground-truth layers (sanity check, trajectory, deliverable, verifiers) are not.

Task Prompt

The verbatim prompt shown to the model — context, the analytical task, hard constraints, and the exact output-format spec. Every task is ≥30 min of real professional work.

Sanity Check

The cognitive trap embedded in the task. Documents the Lazy-AI failure mode and the Expert path around it — not shown to the model.

Golden Trajectory

Extracted from the SME's solution logic by an LLM: the deterministic reasoning path in two layers — named subtasks (Layer 1) and tool/source-annotated steps (Layer 2) with golden answer ranges.

Golden Deliverable

The expert-level output, also LLM-extracted from the solution logic: concrete numbers in the exact format the prompt prescribes, not a description.

Scored Verifiers

Atomic, binary pass/fail criteria grouped by subtask. Each has a specific value, a dependency on its parents, and a DAG weight.

Dependency Graph

Constructed automatically from the golden trajectory: an algorithm identifies root verifiers, wires the minimal parent-child dependencies, validates acyclicity, and assigns depth-proportional weights — turning a pile of failures into a cascade story (one root error vs. many independent ones).

03Dependency graph and DAG weighting

The graph is built automatically from the golden trajectory: an algorithm identifies the root verifiers (values from a single source), wires the minimal parent-child dependencies needed to compute each non-root verifier, validates acyclicity, then computes each verifier's transitive ancestors. Weights follow from that structure — depth-proportional, so the deeper in the chain a verifier sits, the more it is worth, and no verifier is ever negligible.

wᵢ = ( 1 + |ancestors(vᵢ)| ) / Σⱼ ( 1 + |ancestors(vⱼ)| ) × 100%

Reading the weight

A root verifier (no parents) gets weight 1/total — small but never zero. A verifier sitting on 10 ancestors gets 11/total. The scheme is invariant to whether dependencies are written directly or transitively.

Independent scoring, no double-penalty

Each verifier is scored on its own. A child can PASS even when its parent FAILs, provided its method is correct given whatever input was used — so a single upstream error doesn't get punished all the way down the chain.

The per-task Dependency graph tab visualises this: node colour marks role (root, calculation, trap, high-weight, recommendation, format) and the % label is the DAG weight.

04Scoring — VPR, difficulty bands, and why not Pass@k

A task's verifiers aren't independent — they form a dependency chain. We score each run by its DAG-weighted Verifier Pass Rate and report the mean across 3 runs as the primary metric.

Verifier Pass Rate (VPR)

The share of a run's verifiers that pass, each weighted by its DAG weight. A run that nails the deep, high-weight chain scores far above one that only clears shallow root checks. We normalise over the verifiers actually scored in that run.

Why mean VPR, not Pass@3

Pass@k assumes criteria are independent. With a dependency chain, a downstream verifier can "pass" in a run where its parent failed — the method was right but the input was wrong. Pass@3 stitches such disconnected wins into a composite run that never existed. Across these tasks Pass@3 adds only ~0.4% over the best single run — failures are structural, not stochastic (full breakdown in the final section).

Difficulty bands

Thresholds adapted from the classical item-difficulty (p-value) index, recalibrated for multi-verifier agentic tasks.

Mean DAG-weighted VPRDifficultyInterpretation

05Study design

Model

Qwen 3.6 27B, open weights, served via OpenRouter with provider pinning. Standardised sandbox tools (openpyxl, python-docx/pptx, pdfplumber, pandas) for file reading.

Runs

3 independent runs per task. Each run's full output is scored by a domain SME against the golden verifiers — binary pass/fail with a written justification per verifier.

Temporal pinning

Every numeric datapoint in a task is pinned to a fixed date, never "current"/"latest", so scoring is stable over time and reruns stay comparable.

08Scaling up

Scaling beyond the pilot rests on two levers: an existing task inventory and pipeline automation. We already hold a repository of 70 tasks from a prior study built on a similar two-layer evaluation design — binary verifiers plus an SME rubric — which can be converted with LLM assistance to align with this study's schema, giving a substantial head start on volume without authoring from scratch.

To hold quality constant as volume grows, an automated pipeline pre-checks the verifiers that are deterministically coverable — exact values, format compliance, presence tests — so SME annotation effort concentrates on the judgement-heavy criteria rather than mechanical ones. Human QC is accelerated the same way: the pipeline surfaces likely discrepancies for review, leaving experts to adjudicate rather than scan every verifier by hand.
